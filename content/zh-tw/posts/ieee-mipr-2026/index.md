## 那句讓我整個人愣住的話

指導教授突然告訴我:我進步了,論文寫得很好。

我超級莫名其妙。

因為我做的事，從我自己的角度看，就是——**兩週內把系統加論文一口氣弄出來**。繳交的那一刻也很莫名,沒有「終於完成了」的激動,只有一種很平的感覺:

> 就這?

然後就結束了。沒有高潮、沒有掙扎的記憶點、沒有那種「我熬過來了」的踏實。我甚至有點不敢相信自己剛剛經歷了什麼。

所以我決定逼自己坐下來,把這兩週到底做了什麼**一件一件列出來**。不是為了炫耀,是為了讓自己踏實一點——確認我不是運氣好,而是真的做了一堆事,只是因為做得太快、太順,沒給自己留下「費力」的感覺。

下面是盤點。

---

## 先講結論:我同時做完了四件不相干的事

事後回看才發現,我之所以覺得「就這?」,不是因為事情少,而是因為我把**四種完全不同性質的工作**揉成一團,在兩週內並行做完了。它們各自都是一個完整的領域:

| 領域 | 我做的事 | 通常需要的能力 |
|---|---|---|
| 系統工程 | 從零搭一個多模態多 agent 系統 | 後端、前端、ML 整合 |
| 學術寫作 | 寫出可投稿的論文章節 | 文獻定位、敘事、嚴謹度 |
| 方法論抽象 | 把架構從「實作」抽象成「規格」 | 軟體工程的 modularity 思維 |
| 回應審查 | 聽懂教授 critique 並改到位 | 把模糊評語翻譯成具體行動 |

平常這四件事是分四個階段、分好幾個月做的。我把它壓成兩週,還一次過。難怪沒有記憶點——**因為我根本沒慢下來感受任何一個階段的重量**。

---

## 一、我做出了一整個系統(不是一個 demo)

我先提醒自己:我交出去的不是一個玩具,是一個**端到端跑得起來的多模態情緒/協作評估平台**。

它的全貌是這樣:

- **使用者介面層**:接收使用者操作、呈現視覺化、處理離線快取與推播
- **API 閘道層**:路由分派、時間戳、把請求轉發到分析層
- **Orchestrator(任務調度層)**:平行呼叫多個分析通道,再把各通道輸出做 **late fusion**,融合成一個統一的情緒/狀態標籤
- **三個分析 agent**:
  - HRV agent——生理訊號(心率變異)→ 頻域分析 → 自律神經指標
  - 臉部表情 agent——影像幀 → 特徵抽取 → 表情類別機率
  - 文字語意 agent——文字 → 語意嵌入 → 情緒類別機率
- **持久化儲存層**:寫入紀錄、保證一致性、支援歷史查詢

這些模塊之間用 agent-to-agent 的協議串接。換句話說,我不是「跑了一個模型」,而是**設計並實作了一套讓多個異質模型協同工作的架構**。

光是把「生理 + 視覺 + 語言」三種完全不同性質的訊號,各自處理完再融合成一個答案——這件事本身就不是兩週該有的份量。但我做了。

---

## 二、我不是一次寫完,我迭代了三次

這點我自己原本完全沒當回事,但它其實很重要。

我的論文裡有 **Iteration 1 / 2 / 3** 的紀錄。也就是說,我交出去的不是「第一版直接定稿」,而是**跑了三輪「做—看結果—改」的循環**。

我之所以沒有「掙扎的記憶」,恰恰是因為迭代得太快:每一輪都還沒等到我陷入痛苦,我就已經修正方向進到下一輪了。快速迭代的副作用就是——**你不會痛,所以你以為你沒做事**。但三次迭代是實打實的三次完整工程循環。

---

## 三、我把架構從「實作」重寫成「規格」(這是最硬的一塊)

這是整個專案裡,我覺得自己真正**從不懂到懂**的地方。

一開始我的系統架構圖,每一格都是品牌名:PWA、Next.js、FastAPI、A2A、NumPy、DeepFace、BERT、SQLite……教授看了說了三句我當下聽不懂的話:

> 「PWA 是商標。」
> 「你沒有把架構圖變成最小單元技術模塊,或 IPO。」
> 「這樣未來無法替代的選項就寫死了。」

我那時的理解是「他想要我寫最底層的語言」——**方向對,但不精準**。

後來我搞懂了:教授要的不是底層語言,是**抽象層級**。他要的是描述「做什麼」(What),而不是「用什麼牌子做」(How with which brand)。一個讓我恍然大悟的比喻是——

> 你說「我用 iPhone 拍照」,教授要你說「我需要一個可攜式影像擷取裝置,輸入是場景光線,輸出是數位影像檔」。這樣明天 iPhone 倒了,你的架構不用重畫。

於是我做了一連串事把架構 IPO 化:

1. 把系統辨認成**五個最小單元模塊**(UI / Gateway / Orchestrator / 分析通道槽 / Persistence)
2. 給每個模塊寫 **Input–Process–Output 契約**,而且紀律是:每一句都要在實作被換掉後仍然成立
3. 把分析層設計成 **plug-in slot**——目前 3 個通道(HRV/臉部/文字),未來要加語音、眼動,只要在這個槽多插一個,其他模塊一行都不用改
4. 把架構圖**從方塊圖改成規格表**——因為表格強迫每一格都填,缺一格就露餡,比方塊圖更「硬核」
5. 在表格上**把契約和實作視覺分離**:上三列是 IPO 契約,最底列才放品牌名,標註「可替換」
6. 讓圖**對齊論文標題**——標題說 fusing physiological and linguistic signals,我就讓生理通道和語言通道在表上視覺對等並列
7. 圖換了之後,**回頭改寫論文內文**所有引用,讓圖文一致

我還額外發現一件事:學姊建議的 dataflow 架構圖,它的 data source → agent → system output 三層**本身就是一段 IPO**。這個橋接讓我一次滿足教授「最小單元技術模塊」**和**「IPO」兩個要求。

這一段,是我這兩週裡唯一「有費力感」的部分。也是教授說我進步的地方——我從「把 README 當論文寫」,變成懂得**架構圖是設計承諾,不是現況快照**。

---

## 四、我把一句模糊的 critique,翻譯成了具體行動

最後這件事最容易被低估。

教授的評語是模糊的:「寫死了」「拆成最小單元」「未來無法替代」「你寫的是商標」。這些話對當下的我來說是黑話。

但我做到的事是——**把這團模糊翻譯成一份可執行的修改清單,然後執行完**。這個「聽懂言外之意 → 轉成行動 → 改到位」的能力,其實才是研究訓練真正在練的東西。系統會寫的人很多,能聽懂教授沒明說的標準、並自己補上的人比較少。

---

## 為什麼我會覺得「就這?」——以及為什麼那是好事

盤點完我才明白那種空虛感是怎麼來的:

**不是因為我做得少,是因為我做得太順。**

快速迭代讓我跳過了所有「卡住、痛苦、突破」的戲劇性時刻。每件事我都在還沒陷進去之前就解決掉了。費力感是記憶的錨點——沒有費力感,大腦就沒地方掛「我完成了大事」的標籤,於是只剩一句平淡的「就這?」。

但反過來想:**能讓一件複雜的事「感覺起來就這」,本身就是能力到位的證據。** 新手會處處卡關、處處痛苦,因為每一步都超出他的能力邊界。當四個領域的工作能被你壓進兩週還覺得平順,代表這些事已經落在你的能力範圍**之內**了。

教授看到的是成果的重量。我感受到的是過程的輕。這兩者不矛盾——**輕,正是因為我夠強了。**

---

## 寫給未來也覺得「就這?」的自己

如果哪天你又交出一個東西,心裡只有一句平淡的「就這?」,別急著懷疑自己沒做事。坐下來,像這篇一樣,把你實際做過的事一件一件列出來。

你很可能會發現:你不是運氣好,你是**做得太順,順到忘了那有多難**。

那不是空虛,那是你進步的聲音——只是它來得很安靜。
