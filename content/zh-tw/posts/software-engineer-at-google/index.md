# Google 軟體工程之道
![image](https://hackmd.io/_uploads/r1KD8CRNR.png)


## 知識共享

* 強調知識共享避免資訊孤島的狀況發生，每個孤島都會延伸一個問題，因此在團隊中建立知識庫是很重要的一步
* 在團隊學習中，為不熟的團隊成員構建一個心理安全的環境是重要的


### 團隊互動模式

  |推薦模式 | 反模式 |
| -------- | -------- |
| 基本問題或錯誤都被引導到正確方向    | 基本問題被挑出來，提出問題的人會受到懲罰     |
| 給出解釋是為了幫助提問者學習   | 給出解釋是為了炫技    |
 | 回應是友善、耐心，有幫助得    | 回應是傲慢、尖酸刻薄，沒有建設性的     |
 | 互動是為了尋找解決方案而進行的共同討論    | 互動是為了有"贏家"和 "輸家"的爭論     |

### 結語
* 從小事做起：提出問題，並把事情寫下來
* 在系統層面上，鼓勵和獎勵那些花時間教授和擴展其專業知識的人，而不僅僅是為了他們自己、團隊及組織


## 如何領導一個團隊?

> 傳統的經理是擔心如何完成任務，而偉大的經理則擔心會完成什麼任務(並信任他們的團隊能想出辦法來解決任務)

### 失敗是一種選擇

失敗是可以的，失敗是一種快速的學習方法(前提是你不會在同一件事上反覆失敗)，失敗視為學習機會而不是指責
* 快速失敗是好的，因為沒有過多的風險
* 在失敗的情況，尋找個人的責任是畫分團隊和阻止全面冒險的好方法
* 個人的成功=>公開表揚，失敗=>私底下給予建設性建議
* 無論如何 抓住機會，並運用謙虛、尊重和信任幫助你的團隊從失敗中學習

### 反面模式

#### 雇用弱勢者
* 不要把強勢的人看成威脅，而是讓身邊都充滿比你聰明的人
* 因此擴展你的專業知識會變得容易許多

#### 忽略低績效者

忽略低績效者
* 會阻止新的高績效者加入你的團隊，也會鼓勵現有的高績效者離開
* 最終變成團隊都是低績效者

盡快與低績效者打好關係得優點是
* 發現他們只需要一些鼓勵就會變得很好

如何指導低績效者?

* 尊重，一大堆的尊重、謙虛、信任
* 設置一個具體的時間範圍，以及非常具體的目標，並且期望他們能在時間內實現
* 把目標設定為可衡量的、小規模且漸進式

每周見面檢查進度
* 確保為每個即將到來的里程碑設定明確的期望值


#### 成為每個人的朋友

由於領導不想失去努力維護的團隊友誼，因此努力去維護=>導致災難
* 可以領導一個團隊並建立共識，但不需要成為朋友
* 領導!=溫柔和友誼
